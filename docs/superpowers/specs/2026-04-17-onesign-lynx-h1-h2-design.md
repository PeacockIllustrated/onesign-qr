# OneSign – Lynx: Horizon 1 + Horizon 2 Design

**Date:** 2026-04-17
**Status:** Design approved — pending implementation planning
**Scope:** Rename + reposition the product, introduce a B2B organisation model, ship horizon-1 business features, and ship horizon-2 hub features (signatures, review funnel, shopfront, super-admin dashboard).

---

## 1. Context and positioning

### Product rename

The product in this repository is being renamed from **"OneSign QR Generator"** to **"OneSign – Lynx"** (en-dash, spaces). OneSign is the parent/company brand. Lynx is the product name. This document does not rename the repository directory (`onesign-qr`), the package name (`onesign-qr-generator`), database schemas, or environment variables — those are deliberately out of scope and can stay as they are.

The name was chosen because "Lynx" is a homophone for "links," fitting the link-aggregation core, and the `OneSign – X` lockup leaves room for future sibling products without the Lynx product expanding to absorb every feature.

### Positioning shift

Lynx is positioned as a **digital-presence platform for small and medium businesses**, explicitly not a creator-first Linktree competitor. The differentiating wedge is that businesses require features creators do not: multi-page presence, team logins, NFC-programmable physical merchandise, review funnels, email signatures, lead-capture inboxes, and custom domains. Creators can use Lynx, but the product surface is designed against SMB needs first.

The features referred to in earlier discussion as "OneSign – Signal" (email signatures) and "OneSign – Boards" (physical merchandise) are **not sub-brands**. They are features within Lynx.

### Current state (what exists today, before this design lands)

The repository already ships:

- QR code generation with managed redirects, styling, multi-format export, and analytics.
- A bio-page builder with a grid-canvas block editor, four starter templates (creator / business / event / portfolio), preview mode, per-block style overrides, and custom favicons/avatars.
- Block types: links, contact cards, image galleries, contact forms, payment links, countdown timers, Google Maps / Spotify / YouTube embeds.
- An analytics dashboard, email+password auth via Supabase, and mobile-optimised layouts.
- Contact-form submissions wired to Resend for owner-notification emails.

This is effectively Lynx v0. Every feature in horizons 1 and 2 is additive to what exists.

### Customer model

Lynx is built for option **B** from the brainstorming: **businesses with multiple staff logins**. One login per business (option A) was rejected for creating technical debt that hurts when real customers ask for multi-user access. Agencies managing multiple businesses (option C) is supported *incidentally* by allowing a user to belong to multiple organisations, but is not a first-class workflow in these two horizons.

### Critical production constraint

Printed QR codes generated by this application are in production use by businesses. The `/r/{slug}` redirect handler is a **life-support path**, not a feature. Every design and migration decision in this document treats unbroken redirects as an inviolable invariant. See Section 6 for the production-safe migration protocol.

## 2. Foundation: the organisation model

Everything in horizons 1 and 2 depends on this foundation. It must ship before any horizon-1 feature.

### New tables

- **`organizations`** — id, name, slug, business profile fields (phone, address, hours JSON, website, logo_url, default_timezone), created_at, stripe_customer_id (populated when the org first uses the shopfront in H2).
- **`organization_members`** — org_id, user_id, role (`owner | admin | member`), invited_by, joined_at.
- **`organization_invites`** — org_id, email, role, token, expires_at, accepted_at.
- **`platform_admins`** — user_id. Deliberately separate from `organization_members` so platform-level super-admin access cannot be granted or escalated via any org-level action.

### Role semantics

- **owner** — full control including billing, org deletion, member management.
- **admin** — manages pages, QR codes, and members; cannot manage other admins, the owner, or billing.
- **member** — edits pages and QR codes, views analytics. Cannot manage people or billing. Members can edit everything the org owns; per-resource ACLs (e.g., "Sarah can edit page A but not page B") are explicitly out of scope.

### Migration from the current user-centric model

Every existing row in `auth.users` gets a personal organisation created automatically, named from the email prefix (e.g. `"peacock's workspace"`), with the user inserted as `owner`. Every existing row in `bio_link_pages`, `qr_codes`, and related tables gains a nullable `org_id` column, backfilled from the user's personal org. The `UNIQUE(owner_id)` constraint on `bio_link_pages` is dropped so an org can have multiple pages.

`owner_id` columns are retained throughout horizons 1 and 2 as a safety belt. They can be dropped in a post-H2 cleanup migration once at least 30 days of zero regressions have passed.

### RLS rewrite

Every policy of the form `auth.uid() = owner_id` changes to:

```
EXISTS (
  SELECT 1 FROM organization_members
  WHERE org_id = <row>.org_id AND user_id = auth.uid()
)
```

The redirect handler at `/r/[slug]/route.ts` uses `createAdminClient()` which bypasses RLS entirely. **This decouples redirect reliability from the RLS rewrite** and is the key lever that makes the migration safe (see Section 6).

### Active organisation and session

A user can belong to many organisations. The active org is stored in a secure HTTP-only cookie and validated server-side on every request against `organization_members`. An org switcher component is added to the dashboard top navigation. JWT contents remain minimal — role is read from `organization_members` per request so permission changes take effect immediately.

### Invite flow

An org owner or admin enters an email and a role. A row is written to `organization_invites` with a random token. Resend sends an invitation email. The invitee clicks the link:

- If they have no account, they are routed through signup and then land inside the org with the invited role.
- If they have an account, they are prompted to accept; on acceptance, a `organization_members` row is inserted and the invite is marked accepted.

### Explicit non-goals for the foundation

- No SSO / SAML. Enterprise-tier work, not in horizons 1–2.
- No OAuth social login. Nice-to-have, not foundational.
- No per-resource ACLs beyond role-at-org-level.
- No activity feed or audit log for org-level actions in H1. (Platform-level audit log for super admin actions *is* in H2 — see Section 4.)

## 3. Horizon 1: the business core

Five feature blocks. All depend on the foundation from Section 2.

### 3.1 Multi-page per organisation

The `UNIQUE(owner_id)` constraint on `bio_link_pages` is dropped. A "Pages" list view is added to the dashboard, showing every page in the active org as a grid of thumbnails using the existing preview renderer. Each page retains its own slug, template, blocks, and style. QR codes can point at any page in the active org.

### 3.2 NFC support

NFC writes a URL; mechanically, the same URL that encodes a managed QR code is also NFC-writable. No new DB object is required. Changes:

- A `medium` enum column is added to `qr_codes`: `qr | nfc | both`. This is purely analytics and shopfront metadata — it does not affect the redirect handler.
- A "Write to NFC" helper view is added to the QR detail page: copies the managed URL, shows step-by-step instructions for common NFC writer apps, and provides a QR-free "URL only" view for customers who want to program tags themselves.
- Scan analytics capture medium via a URL parameter (`?m=nfc`) when the URL is baked into shopfront-fulfilled NFC merchandise.

Web NFC in-browser writing is explicitly out of scope (Android-Chrome only, flaky). Lynx designs for the managed URL, which is the medium-independent source of truth.

NFC is a **marketed feature** on the landing page, not a hidden power feature. This influences design polish requirements on the "Write to NFC" flow.

### 3.3 Lead capture routing and inbox

Today, contact-form submissions are emailed to the bio page owner and stored in `bio_form_submissions`. H1 extensions:

- **Per-page email routing config**: submissions can be sent to one or more email addresses per page, stored as JSON on the page.
- **Submissions inbox** in the dashboard — list view per page, detail view, read/archived state, date-range filter, full-text search over submission content.
- **CSV export** of all submissions scoped to a date range.
- **Spam mitigation**: retain existing honeypot + rate limit. No CAPTCHA in H1.
- **Slack and generic outgoing webhook routing is deferred to H2.** Email-only in H1.

### 3.4 Custom domains per page

Bio pages gain a `custom_domain` field with a verification state enum (`pending | verified | active`). Verification is via CNAME pointing at a platform-controlled hostname. SSL certificates are provisioned automatically via Vercel's Domains API.

Limits: one custom domain per page, no wildcard domains in H1.

**Custom domains are paywalled.** They are a Pro-tier feature. The tier system (see Section 3.5) is built to gate this.

### 3.5 Pricing tier system (lightweight)

A `plan` enum column is added to `organizations`: `free | pro`. A feature-flag helper `orgHasFeature(org, feature)` is used at both the UI layer (to hide/show) and the API layer (to enforce). In H1, the only gated feature is **custom domains**.

Full pricing strategy — price points, annual discounts, per-seat billing, usage metering — is a separate brainstorm. H1 only needs the plumbing. Stripe subscription billing for the Lynx app itself is implemented as part of the H2 shopfront Stripe work, not H1.

### 3.6 Analytics upgrades

Small additions to support the H1 story:

- Medium tag (`qr | nfc`) captured on scan events when present.
- Block-level click tracking completed on bio pages (partially present today).
- Form-submission counts shown alongside scan counts on the page dashboard.

### Explicit non-goals for H1

- No scheduled publishing.
- No A/B testing of pages.
- No activity feed of who-did-what within an org.
- No approval workflows.
- No Slack or webhook lead routing (→ H2).

## 4. Horizon 2: hub features and shopfront

Four feature areas. The super-admin dashboard is a prerequisite for the shopfront.

### 4.1 Email signature generator

A new dashboard area titled **Email Signatures**.

- **Templates**: 4–6 starter layouts (minimal, photo-left, logo-bar, banner, compliance-heavy with disclaimer, appointment-link).
- **Data sources**: organisation business profile (logo, phone, address) for brand fields; per-user fields (name, title, direct email, photo) for personal fields.
- **Per-user / per-org split**: an org admin picks a template and locks the brand (logo, colours). Each member fills in their own personal fields and generates their own signature against that locked brand.
- **Outputs**:
  - **HTML string** for pasting into Gmail / Outlook / Apple Mail.
  - **Hosted signature URL** — a public page rendering the signature, as a fallback for clients that strip HTML and as a "scan my signature" target for printed marketing.
  - **PDF export** via `@sparticuz/chromium` + `playwright-core`, rendering the hosted signature page to PDF. Serverless-compatible, no new infrastructure required.
- **QR/NFC integration**: any signature can optionally embed a QR for the user's bio page or the org's main page, providing natural cross-sell back into the core product.

PNG export is explicitly out of scope. PDF covers the printable-asset use case better.

### 4.2 Review funnel block

A new bio-page block type. When scanned or tapped, the customer sees a star rating prompt.

- 5 or 4 stars → redirect to the business's configured Google Review URL.
- 3 stars or below → open a private feedback form; the submission flows to the lead capture inbox.
- Optional: redirect target is configurable per page (Google / TripAdvisor / Trustpilot / Facebook).

This is a compact feature (~one week of work) but a category-defining one for SMBs. It reuses the bio-block rendering pipeline and the existing form-submissions plumbing.

### 4.3 Shopfront

The largest H2 work area. Three sides: customer-facing (the org buying merchandise), the product model, and fulfilment.

#### 4.3.1 Customer-facing experience (`/app/shop`)

Inside the dashboard, browsed and purchased by the active org:

- Category filter (NFC cards, review boards, table talkers, window decals, badges).
- Product detail with variants (size, colour, quantity) and a **live preview** showing the product with the org's logo and a selected bio page's URL or QR baked in.
- Checkout via **Stripe Checkout** (hosted, not custom). OneSign is the seller of record; no Stripe Connect.
- **Stripe Tax** for automatic tax calculation. Shipping rates configurable per product.
- Post-purchase: order confirmation email; shipping email when the super admin marks the order shipped; optional delivery email.

#### 4.3.2 Product model

- **`shop_products`** — sku, name, description, base_price, media, is_active, category, created_by_platform_admin.
- **`shop_product_variants`** — product_id, label, sku, price_delta, stock_qty (nullable = made-to-order / unlimited).
- **`shop_product_customizations`** — product_id, field_type (`logo | bio_page_link | custom_text | color`), required, constraints JSON. Drives both the live preview and validation at checkout.
- **`shop_orders`** — org_id, user_id, stripe_session_id, status (`pending | paid | in_production | shipped | delivered | cancelled | refunded`), shipping_address JSON, totals, created_at.
- **`shop_order_items`** — order_id, product_id, variant_id, customization_values JSON (the actual logo URL / bio page URL / custom text the customer picked), quantity, unit_price, line_total.

#### 4.3.3 Fulfilment (v1: manual, in-house)

"Fulfilment" is a kanban board in the super-admin dashboard: new orders → in production → shipped → delivered. The super admin uploads proof-of-production images, enters a tracking number, and clicks "Mark shipped," which fires the shipping email. No integration with print-on-demand vendors in H2.

#### Shopfront non-goals

- No marketplace. Only OneSign lists products.
- No multi-currency. **GBP only.**
- No print-on-demand vendor integration.
- No subscription-billed shop products. One-off purchases only. (Subscription billing for the Lynx app tier system is a separate feature, implemented alongside Stripe integration.)
- No product reviews / UGC ratings on shop products.

### 4.4 Super-admin dashboard (`/admin`)

A separate area of the application, only accessible to users whose `user_id` exists in `platform_admins`. Completely different information architecture from the org-facing app.

#### Screens

- **Overview** — platform KPIs (signups, active orgs, MRR once billing is live, orders this week, form submissions, scans).
- **Product catalog CRUD** — create / edit / archive products and variants. Upload media. Configure customization fields. Preview what customers will see.
- **Orders** — full orders list, filter/search, detail view, fulfilment actions, refund button (fires Stripe refund), internal notes.
- **Organisations** — list all orgs, search, view org detail (members, pages, QRs, plan, billing status), soft-delete, flag as suspicious.
- **Users** — list all users, search, view user detail.
- **View-as mode** (read-only) — a super admin can render any org's dashboard in read-only mode to reproduce a customer's view for support. A banner is shown throughout the view-as session. **No write actions are possible in view-as mode.** Full impersonation (including write access) is explicitly deferred to a post-H2 decision — ship view-as now, add full impersonation when support volume justifies the security cost.
- **Platform audit log** — every super-admin action recorded: who did what, when, to which org or user. View-as entry and exit events logged.
- **Feature flags / config** — edit global platform settings (free-tier limits, featured products on the shop).

#### Security

- `/admin` is blocked at Next.js middleware before any data loads — no risk of rendering the page and hiding buttons.
- **Step-up auth** on entry: re-enter password even if the main app session is live. Short idle timeout on the admin session (30 minutes).
- All super-admin actions audit-logged. View-as entry and exit logged with reason field.

Phase 2 is long because shopfront, Stripe, admin dashboard, and signatures are four substantial features. If the date needs to be pulled in, the planned de-scope is **signatures → Pro-only-HTML** (drop the hosted URL and PDF export from the first shopfront ship).

## 6. Production-safe migration protocol

### Why this section exists

Printed QR codes generated by this application are in active production use by businesses. If the `/r/{slug}` redirect handler flickers during migration, real customers lose revenue and trust. This section codifies the protocol that makes the Phase 0 migration safe.

### Invariants (non-negotiable)

1. **Zero redirect failures.** Every existing printed QR must resolve to the identical `destination_url` at every moment during migration.
2. **Zero data loss.** Row counts match pre- and post-migration for every existing table.
3. **Reversible at every step.** Every phase has a one-command rollback to a known-good state.
4. **No user downtime on redirects.** `/app/*` may briefly degrade; `/r/{slug}` may not.

### Key architectural lever

The redirect handler at [src/app/r/[slug]/route.ts:20](src/app/r/%5Bslug%5D/route.ts#L20) uses `createAdminClient()` which bypasses RLS entirely. **This decouples the RLS rewrite from redirect reliability.** The only way to break redirects is to break the schema on the columns the handler reads: `slug`, `destination_url`, `is_active`, `analytics_enabled`, `mode`. The migration protocol therefore treats these columns as frozen.

### Strategy: Expand → Migrate → Cutover → Contract

Each phase is independently deployable and independently reversible.

#### Phase 0.A — Expand (additive only, zero behavioural change)

- Add new tables: `organizations`, `organization_members`, `organization_invites`, `platform_admins`.
- Add nullable `org_id` column to every table that currently has `owner_id`.
- Do not change RLS. Do not change application code. Do not drop or rename anything.
- This phase is fully backward compatible. Deploy, wait 24 hours, confirm nothing is on fire, then proceed.

#### Phase 0.B — Backfill (data only, no schema change)

- For every existing `auth.users` row, create a personal organisation and insert `(org_id, user_id, 'owner')` into `organization_members`.
- For every existing `bio_link_pages`, `qr_codes`, and related row, populate `org_id` from the owner's personal org.
- Execute inside a transaction; on any failure, nothing changes.
- **Verification gate before proceeding**:
  - `COUNT(*) WHERE org_id IS NULL` returns 0 on every affected table.
  - `COUNT(*) FROM auth.users` equals `COUNT(DISTINCT user_id) FROM organization_members`.

#### Phase 0.C — Cutover (per-table, feature-flagged)

- Rewrite RLS on the **least critical table first** (`bio_form_submissions`). Deploy. Monitor for 24 hours.
- Move to the next table. Repeat.
- End with `qr_codes`. The redirect handler is unaffected (bypasses RLS) — the RLS rewrite on `qr_codes` only affects the authenticated `/app/*` experience.
- Each per-table cutover is wrapped in a feature flag so rollback is a single environment variable change, not a migration.
- **Dual-write window**: during and for two weeks after cutover, writes populate both `org_id` and `owner_id`. Reads gradually move to `org_id`. This keeps rollback cheap.

#### Phase 0.D — Contract (drop old columns, ≥30 days after cutover)

- After a full month with zero regressions, drop `owner_id` columns and their associated indexes in a separate migration.
- Not strictly required for subsequent feature work. May be deferred indefinitely.

### Pre-flight checklist (must be all green before any migration runs)

- [ ] Prod database backup verified: restore-to-staging test completed within the last 7 days.
- [ ] PITR (point-in-time recovery) confirmed active on the Supabase project.
- [ ] Full staging rehearsal: prod database clone restored to staging, the full Expand → Migrate → Cutover → Contract sequence executed end-to-end with automated verification passing.
- [ ] Redirect synthetic monitor deployed: pings a known-good production QR every 60 seconds and alerts on any non-307 response.
- [ ] Slug integrity diff report: a script exports `(slug, destination_url, is_active)` for every `qr_codes` row pre-migration, re-exports post-migration, and must return zero diffs.
- [ ] Row-count diff report for every affected table: pre and post counts must match.
- [ ] Rollback runbook written, reviewed by someone other than the author, and rehearsed on staging.
- [ ] Customer communication drafted: even though zero downtime is expected, active orgs get ≥48 hours notice with what to expect and a support channel.
- [ ] Migration window scheduled in the lowest-traffic slot (Sunday 03:00–05:00 BST based on existing customer base) with an on-call person awake.

### Ongoing safeguards (not just migration day)

- **Redirect handler regression tests**: golden-path integration tests hit a real test Supabase with scenarios for every combination of `mode × is_active × analytics_enabled`. Run on every PR that touches `qr_codes`, the redirect route, or the admin Supabase client.
- **Redirect latency SLO**: p99 under 200ms. If the migration introduces a slow query path, the monitor surfaces it.
- **Schema-lint CI check**: a test that fails CI if any proposed migration drops or renames the columns `slug`, `destination_url`, `is_active`, `analytics_enabled`, `mode` on `qr_codes`. Hard gate.

### What "perfect execution" looks like

- Zero redirect failures during or after migration (synthetic monitor log confirms).
- Zero data loss (row-count diff report empty).
- Zero duplicate orgs (every `auth.users` row maps to exactly one personal org).
- All existing slugs resolve to identical destination_urls (slug-integrity diff report empty).

## 8. Glossary

- **Active org** — the organisation whose data the logged-in user is currently operating on. Selected via the org switcher; persisted in a secure cookie.
- **Lynx** — the product name. The full lockup is "OneSign – Lynx" (en-dash, spaces).
- **Managed URL** — a URL of the form `/r/{slug}` that the application controls and can change destinations for. The core promise of the QR side of the product.
- **Medium** — the physical delivery mechanism for a managed URL: QR code, NFC tag, or both. A metadata field, not a behavioural change.
- **Personal org** — the auto-created organisation given to every existing user during Phase 0.B migration so that the user's existing data has a valid `org_id`.
- **Platform admin / super admin** — a user with a row in `platform_admins`, granting access to `/admin`. Platform-level, independent of any `organization_members` role.
- **View-as mode** — a read-only super-admin capability to render an org's dashboard as the customer sees it, for support purposes. Writes are impossible in this mode.
